# Design Risk Register

This register records security-relevant design decisions, reasons, tradeoffs,
framework mappings, accepted risks, owner roles, status, and follow-up work.
Open gaps remain in `docs/security/governance/security-gap-register.md`. Role definitions,
review cadence, accepted-risk handling, and off-repo ownership rules are in
`docs/security/governance/security-governance.md`.

Category: [Security governance](../README.md#governance).

| Design decision | Reason | Security impact | Tradeoff | Related framework controls | Accepted risk | Follow-up status | Owner role | Status / tracking | Review trigger |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Separate admin/customer deployment boundary | Keep privileged staff/admin flows out of the customer app and use separate cookies, secrets, app mode, database role, and route inventory | Reduces blast radius from customer-route bugs and cookie/session confusion | More deployment and secret-management complexity | OWASP ASVS access control, OWASP API BFLA, CIS account management | Public admin hostname returns denied responses for ACME/edge handling; live private access must be verified | No | Deployment Owner | Partially implemented; external live private-access evidence remains required | Before admin route, Nginx, Tailscale, or deployment-boundary changes |
| TOTP baseline MFA | Approved MFA direction uses authenticator TOTP and one-time recovery codes | Provides MFA without adding high-level authentication libraries or unsupported factor flows | TOTP is phishing-susceptible compared with hardware-bound authenticators | NIST SP 800-63B, OWASP ASVS MFA | Accept TOTP phishing risk for coursework scope; rely on generic errors, rate limits, step-up, and audit | No | Security Owner | Accepted risk; current approved MFA baseline | When MFA policy, reset MFA, or admin authentication changes |
| TOTP-only active MFA surface | Project constraints reject the previous browser-credential experiment and the approved direction is TOTP plus one-time recovery codes | Removes unused credential stores and fallback paths that could diverge from current MFA policy | Stronger phishing-resistant factors are deferred/out of scope | NIST SP 800-63B, OWASP ASVS MFA | The prior implementation and compatibility schema are removed; any new factor requires separate design and migration approval | No | Security Owner | Implemented current policy | If project policy changes or a new MFA proposal is approved |
| PostgreSQL-backed server-side session architecture | Keep cookies opaque and store signed session payloads in application-owned PostgreSQL state | Enables revocation, inventory, HMAC payload integrity, idle expiry, absolute lifetime, and single-active-session replacement | More database dependency than stateless cookies | ASVS session management, NIST SP 800-63B sessions | Cryptographic device-bound proof remains an accepted defense-in-depth item | No | Application Owner | Implemented; bearer-session proof-of-possession remains accepted defense in depth | Before session storage, expiry, revocation, or risk-binding changes |
| Risk-based session context instead of claimed device binding | Public browsers do not expose an application-controlled per-request private key; IP and User-Agent are only risk signals | HMAC-derived coarse network and User-Agent drift can require customer reauthentication or revoke higher-risk/admin sessions | Legitimate network/browser changes can require login, while copied cookies remain bearer credentials when context is reproduced | ASVS session management, NIST SP 800-63B sessions | Cryptographic device-bound proof remains open defense in depth; do not describe this control as proof of possession | Accepted defense-in-depth gap | Security Owner | Accepted risk; context is a risk signal, not proof of possession; Tailscale is the admin private network/device boundary | When session-risk policy or browser/device-key support changes |
| Production startup fail-closed guard | Prevent production traffic when required security prerequisites are unsafe | Blocks weak config for CSRF, secrets, password length, proxy trust, audit/alert state, and database privileges | Misconfiguration causes startup/readiness failure until fixed | CIS secure configuration, MAS Cyber Hygiene | Operational availability can be affected by strict checks; preserve break-glass runbooks rather than weakening guard | No | Deployment Owner | Implemented | Before production guard, WSGI startup, or deployment readiness changes |
| Staff/admin/root separation of duties | Bank-staff business work, admin technical/security work, and root-admin privileged platform work need separate UI, route authorization, and dual control for highest-risk root-admin changes | Reduces lower-role access to audit, alert, invite, staff lifecycle, and manual recovery powers; maker-checker approval reduces single-operator abuse for staff lifecycle and final manual recovery changes; the transaction dispute review queue is gated by a plain-staff-only guard (`require_plain_staff_session`) that excludes `admin`/`root_admin`, who retain only audit-log oversight | Staff business-operation routes that do not yet exist beyond the transaction dispute queue are shown only as disabled placeholders and must be implemented separately; highest-risk root-admin changes require a second root admin | OWASP ASVS access control, CIS account management, MAS access control | No customer support or fraud review admin route beyond the transaction dispute queue is claimed until it exists and has route inventory coverage | No | Application Owner / Security Owner | Implemented | Before admin dashboard, staff workflow, invite, staff lifecycle, audit viewer, alert, dispute queue, or manual recovery route changes |
| Admin route inventory testing | Make admin route authorization explicit and separate from customer route policy | Prevents silent privileged route exposure | Route additions require test updates | OWASP ASVS access control, OWASP API inventory | None for current route surfaces | No | Application Owner | Implemented | Whenever admin routes, decorators, or admin services change |
| Audit hash/HMAC integrity | Make audit row order and contents tamper-evident | Detects row mutation, tail deletion, chain rewind, and anchor mismatch | Requires protected HMAC key and anchor operations | OWASP ASVS logging, CIS audit log management, MAS monitoring | Live anchor storage and review cadence remain operator evidence | No | Security Owner / External operator / outside repo | Partially implemented; repository controls exist, live anchor handling remains external evidence | Before audit schema, anchor, alerting, or operator-review changes |
| Zero-trust/private admin-staging access direction | Staging needs identity-aware browser access; admin should not be public internet reachable | Cloudflare Access plus server-level AOP reduce origin bypass; safe audit/evidence limits diagnostic leakage; private Tailscale reduces public admin exposure | Requires Cloudflare/Tailscale operator setup and live verification; current global AOP CA is broader than hostname-scoped trust | CIS access control, MAS Cyber Hygiene, OWASP SAMM Operations | Repository automation verifies sanitized provider state and host posture, but live IdP, GitHub Environment, ACL, device, and operator state remain external | Operator verification remains; hostname-scoped AOP is a follow-up | Deployment Owner / External operator / outside repo | Partially implemented external prerequisite; repository controls fail closed and record safe reason codes while provider-owned evidence remains operator-owned | Before Cloudflare, Tailscale, admin exposure, staging access, or provider-logging changes |
| Protected GitHub-hosted private-admin verification | A trusted self-hosted tailnet runner is unavailable, but private reachability needs repeatable protected evidence after production deploy | Manual environment approval, no PR trigger, no checkout, an ephemeral tagged identity, explicit OAuth/auth-key selection, and pre-join/private response checks limit exposure | One protected Tailscale credential mode is available to the job; each production release pauses for approval after public TLS | CIS access control, MAS Cyber Hygiene, OWASP SAMM Verification, NIST SSDF Protect | OAuth is the production default; auth-key compatibility requires separate short-lived key governance | No | Security Owner / Deployment Owner / External operator / outside repo | Implemented as a required protected gate with external credential/policy/live-run prerequisites | Before workflow triggers, environment protection, credential mode, tags/grants, private hostname, or runner model changes |
| Dedicated Gitleaks full-history scanning | The custom scanner has intentionally focused patterns, so a mature independent engine adds broader credential coverage | A checksum-verified Gitleaks 8.30.1 CLI scans all refs with redacted output, least privilege, no production secrets, and no uploaded report | Full-history scans cost CI time and require private false-positive triage; branch protection must be configured outside the repository | NIST SSDF Produce, OWASP SAMM Verification, CIS secure configuration | No baseline is accepted; reviewed false positives are constrained by rule, exact path, line shape, and, where historical, commit; future exceptions must be confirmed fake, synthetic, expired, or public non-secret metadata | No | Security Owner | Implemented in `.github/workflows/gitleaks.yml` alongside the custom repository secret scanner | Before Gitleaks version, rules, scope, reporting, permissions, allowlist, or baseline changes |
| Dedicated ShellCheck, Hadolint, and Semgrep gates | Language-appropriate lint and SAST should cover the whole repository rather than a hand-maintained path list | Shared tracked-file discovery prevents silent shell/Dockerfile omissions; local/OSS Semgrep blocks high-confidence ERROR findings without granting a third party source-upload credentials | Style-level shell/container findings are strict; Semgrep lower severities remain review-only to control initial noise | NIST SSDF Produce, OWASP SAMM Implementation and Verification, CIS vulnerability management | Branch protection still has to require the three check names outside the repository; any suppression must be narrow, exact, and justified beside the affected instruction or statement | No | Security Owner / Application Owner | Implemented in `.github/workflows/shellcheck.yml`, `.github/workflows/hadolint.yml`, and `.github/workflows/semgrep.yml` with local parity in `scripts/ci-local` | Before scanner version, checksum/digest, discovery, rule pack, severity, exclusion, suppression, permission, or reporting changes |
| Confirmation-gated Tailscale Serve provisioning | Replace informal host commands with reproducible private-HTTPS automation | Dry-run, explicit confirmation, pinned repository metadata, fixed tag/hostname/backend, pre/post verification, and no committed secret reduce operator error | Live execution can still disrupt node settings; `tailscale up --reset` and Serve changes require an approved maintenance window and rollback access | CIS secure configuration, MAS Cyber Hygiene, OWASP SAMM Operations | ACL/policy application, device approval, group changes, and evidence retention remain manual | No | Deployment Owner / Security Owner / External operator / outside repo | Implemented under `ops/tailscale/`; staging admin is intentionally not configured | Before package source, auth mode, tags, Serve/Funnel, private hostname, or staging-admin changes |
| Read-only EC2 Tailscale admin preflight | Local listener, Nginx, Serve, and Funnel drift need repeatable host evidence without provisioning credentials | A non-mutating script fails closed on unsafe local exposure and complements remote reachability evidence | Normal CI can test only the script contract with stubs; operators must execute live modes on EC2 and retain the result | CIS secure configuration, MAS Cyber Hygiene, OWASP SAMM Verification | Keep provisioning, policy changes, and credentials out of the verifier; `documentation-only` output is never production evidence | No | Deployment Owner / External operator / outside repo | Implemented in `ops/deploy/verify-tailscale-admin-access`; production bootstrap installs it and live execution remains an operator step | Before listener, Nginx, Tailscale, Serve/Funnel, hostname, or admin-edge changes |
| GitHub-hosted runner SSH conflict | GitHub-hosted runners have changing public IPs and should not justify global SSH | Public SSH exposure or brittle source-IP allowlists | Deployment now joins Tailscale with separate staging and production tags before OpenSSH; public port `22` removal remains provider-owned evidence | CIS secure configuration, NIST SSDF Protect | Keep OAuth clients environment-scoped, deny cross-environment paths, retain strict known-host checking, and remove public SSH only after private deploy succeeds | Mitigated; external firewall action required | Deployment Owner / External operator / outside repo | Provider configuration evidence | Before SSH hardening, runner topology, Tailscale policy, or OIDC/SSM changes |
| Reporting-only SonarQube initial rollout | SonarQube Cloud reports coverage, maintainability, duplication, reliability, and security findings; reporting-only is safer until the baseline and false positives are tuned | Avoids breaking releases on an uncalibrated scanner while collecting dashboard evidence | Reporting-only scanners do not enforce fixes by themselves | NIST SSDF Produce, OWASP SAMM Verification | Conditional accepted risk: review the baseline and authorize any blocking quality-gate policy through a separate security review | Baseline review remains | Security Owner | Implemented as reporting-only | If SonarQube scope, project administration, or quality-gate policy changes |
| Encrypted backup helper and restore preflight | Backups contain personal, audit, and banking-like data | Reduces persistent plaintext dump exposure and blocks unsafe restore attempts with owner, mode, parent directory, repository/workspace, target, and identity checks | Requires host-managed age recipients/identity and operator scheduling | PDPA protection, CIS data protection, MAS resilience | Archive pruning remains operator-approved and evidence-owned outside the repository | Backup schedule and restore-drill evidence remain external | Deployment Owner / External operator / outside repo | Implemented tooling; backup schedule, restore-drill, and archive-retention evidence remain external/operator-owned | Before backup schedule, restore, retention, or age-recipient changes |
| PDPA documentation before automation | Document data categories, retention expectations, and incident handling before implementing broad deletion jobs | Avoids unsupported privacy claims and evidence-destroying deletion | Preserved-category disposal remains manual/operator-approved unless a future reviewed scheduler is added | Singapore PDPA, NIST SSDF Respond, CIS data protection | No automatic deletion/anonymization is claimed; approval evidence remains external | No | Documentation Owner / Application Owner | Implemented procedure; future automation requires separate review | Before privacy, retention, deletion, anonymization, or data-category changes |
