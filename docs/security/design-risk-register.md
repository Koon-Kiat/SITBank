# Design Risk Register

This register records security-relevant design decisions, reasons, tradeoffs,
framework mappings, accepted risks, and follow-up work. Open gaps remain in
`docs/security/security-gap-register.md`.

| Design decision | Reason | Security impact | Tradeoff | Related framework controls | Accepted risk | Follow-up issue |
| --- | --- | --- | --- | --- | --- | --- |
| Separate admin/customer deployment boundary | Keep privileged staff/admin flows out of the customer app and use separate cookies, secrets, app mode, database role, and route inventory | Reduces blast radius from customer-route bugs and cookie/session confusion | More deployment and secret-management complexity | OWASP ASVS access control, OWASP API BFLA, CIS account management | Public admin hostname returns denied responses for ACME/edge handling; live private access must be verified | No |
| TOTP baseline MFA | Approved MFA direction uses authenticator TOTP and one-time recovery codes | Provides MFA without adding high-level authentication libraries or unsupported factor flows | TOTP is phishing-susceptible compared with hardware-bound authenticators | NIST SP 800-63B, OWASP ASVS MFA | Accept TOTP phishing risk for coursework scope; rely on generic errors, rate limits, step-up, and audit | No |
| No WebAuthn/passkeys in approved staff/admin flow | Project constraints reject the high-level WebAuthn library and current approved direction is TOTP | Avoids unapproved library/flow while preventing passkey fallback from bypassing TOTP | Stronger phishing-resistant factors are deferred/out of scope | NIST SP 800-63B, OWASP ASVS MFA | Legacy compatibility routes fail closed or are disabled; do not reintroduce passkey/WebAuthn login or step-up without separate approval | No |
| PostgreSQL-backed server-side session architecture | Keep cookies opaque and store signed session payloads in application-owned PostgreSQL state | Enables revocation, inventory, HMAC payload integrity, idle expiry, and absolute lifetime | More database dependency than stateless cookies | ASVS session management, NIST SP 800-63B sessions | Active-session cap and device-bound proof remain open defense-in-depth items | No |
| Production startup fail-closed guard | Prevent production traffic when required security prerequisites are unsafe | Blocks weak config for CSRF, secrets, password length, proxy trust, audit/alert state, and database privileges | Misconfiguration causes startup/readiness failure until fixed | CIS secure configuration, MAS Cyber Hygiene | Operational availability can be affected by strict checks; preserve break-glass runbooks rather than weakening guard | No |
| Admin route inventory testing | Make admin route authorization explicit and separate from customer route policy | Prevents silent privileged route exposure | Route additions require test updates | OWASP ASVS access control, OWASP API inventory | None for current route surfaces | No |
| Audit hash/HMAC integrity | Make audit row order and contents tamper-evident | Detects row mutation, tail deletion, chain rewind, and anchor mismatch | Requires protected HMAC key and anchor operations | OWASP ASVS logging, CIS audit log management, MAS monitoring | Live anchor storage and review cadence remain operator evidence | No |
| Zero-trust/private admin-staging access direction | Staging needs identity-aware browser access; admin should not be public internet reachable | Reduces origin bypass and public admin exposure risk | Requires Cloudflare/Tailscale operator setup and live verification | CIS access control, MAS Cyber Hygiene, OWASP SAMM Operations | Repository can document/test config, but live IdP/tailnet state remains external | No |
| GitHub-hosted runner SSH conflict | GitHub-hosted runners have changing public IPs and should not justify global SSH | Pushes deployment toward allowlisted runner/bastion/VPN or OIDC + SSM | Current SSH workflow needs an approved source path until migration | CIS secure configuration, NIST SSDF Protect | Temporary SSH path is acceptable only with allowlisted source or reviewed time-boxed maintenance | No |
| Reporting-only SonarQube initial rollout | SonarQube is not active in this repository; if introduced, reporting-only is safer until false positives and scope are tuned | Avoids breaking releases on an uncalibrated scanner while collecting findings | Reporting-only scanners do not enforce fixes by themselves | NIST SSDF Produce, OWASP SAMM Verification | Conditional accepted risk: time-box reporting-only mode and add gates only after tuning | No |
| Encrypted backup helper and restore preflight | Backups contain personal, audit, and banking-like data | Reduces persistent plaintext dump exposure and blocks unsafe restore attempts | Requires host-managed age recipients/identity and operator scheduling | PDPA protection, CIS data protection, MAS resilience | Retention/disposal automation for backup archives remains open | No |
| PDPA documentation before automation | Document data categories, retention expectations, and incident handling before implementing broad deletion jobs | Avoids unsupported privacy claims and evidence-destroying deletion | Retention automation remains manual/operator-guided until implemented | Singapore PDPA, NIST SSDF Respond, CIS data protection | Accepted only as documented gap; do not claim automatic deletion/anonymization | No |
